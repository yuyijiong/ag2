title: Code Execution
---

AG2 agents can execute code from a message passed to them (e.g., those containing code blocks) and output a message with the results of the execution for the following agent to interpret.

There are two types of built-in code executors, one is the command line code executor, which runs code in a command line environment such as a MacOS or Linux shell, and the other is a Jupyter executor, which runs code in an interactive Jupyter kernel.

For each type of executor, AG2 provides two ways to execute code: locally and in a Docker container. For development and testing, not recommended for production, you can run it on the same host platform where AG2 is running, i.e., the local operating system. For better The other way is to execute code in a Docker container. The table below shows the combinations of code executors and execution environments.

| Code Executor (`autogen.coding`) | Environment | Platform |
| ---------------------------- | ----------- | -------- |
| [`LocalCommandLineCodeExecutor`](/docs/api-reference/autogen/coding/LocalCommandLineCodeExecutor)  | Shell       | Local  |
| [`DockerCommandLineCodeExecutor`](/docs/api-reference/autogen/coding/DockerCommandLineCodeExecutor) | Shell       | Docker |
| [`RemyxCodeExecutor`](/docs/api-reference/autogen/coding/RemyxCodeExecutor) | Pre-built Docker | Remyx Cloud |
| [`YepCodeCodeExecutor`](/docs/api-reference/autogen/coding/YepCodeCodeExecutor) | Serverless | YepCode Cloud |
| [`DaytonaCodeExecutor`](/docs/api-reference/autogen/coding/DaytonaCodeExecutor) | Sandbox | Daytona Cloud |
| [`jupyter.JupyterCodeExecutor`](/docs/api-reference/autogen/coding/jupyter/JupyterCodeExecutor)           | Jupyter Kernel (e.g., python3) | Local/Docker |

## Local Execution

The figure below shows the architecture of the local command line code executor
([`autogen.coding.LocalCommandLineCodeExecutor`](/docs/api-reference/autogen/coding/LocalCommandLineCodeExecutor)).

<Danger>
Executing LLM-generated code poses a security risk to your host environment.
</Danger>

![Code Executor No Docker](./assets/code-executor-no-docker.png)

Upon receiving a message with a code block, the local command line code executor first writes the code block to a code file, then starts a new subprocess to execute the code file. The executor reads the console output of the code execution and sends it back as a reply message.

Here is an example of using the code executor to run a Python code block that prints a random number.

Before running this example, we need to make sure the `matplotlib` and `numpy` are installed.

```bash
pip install -qqq matplotlib numpy
```

First we create an agent with the code executor that uses a temporary directory to store the code files.

We specify `human_input_mode="ALWAYS"` to manually validate the safety of the code being  executed.

```python
import tempfile

from autogen import ConversableAgent, LLMConfig
from autogen.coding import LocalCommandLineCodeExecutor

# Create a temporary directory to store the code files.
temp_dir = tempfile.TemporaryDirectory()

# Create a local command line code executor.
executor = LocalCommandLineCodeExecutor(
    timeout=10,  # Timeout for each code execution in seconds.
    work_dir=temp_dir.name,  # Use the temporary directory to store the code files.
)

# Create an agent with code executor configuration.
code_executor_agent = ConversableAgent(
    "code_executor_agent",
    llm_config=False,  # Turn off LLM for this agent.
    code_execution_config={"executor": executor},  # Use the local command line code executor.
    human_input_mode="ALWAYS",  # Always take human input for this agent for safety.
)
```

Now we have the agent generate a reply given a message with a Python code block.

```python
message_with_code_block = """This is a message with code block.
The code block is below:
``""" + """`python
import numpy as np
import matplotlib.pyplot as plt
x = np.random.randint(0, 100, 100)
y = np.random.randint(0, 100, 100)
plt.scatter(x, y)
plt.savefig('scatter.png')
print('Scatter plot saved to scatter.png')
``""" + """`
This is the end of the message.
"""

# Generate a reply for the given code.
reply = code_executor_agent.generate_reply(messages=[{"role": "user", "content": message_with_code_block}])
print(reply)
```

```console
>>>>>>>> NO HUMAN INPUT RECEIVED.

>>>>>>>> USING AUTO REPLY...

>>>>>>>> EXECUTING CODE BLOCK (inferred language is python)...
exitcode: 0 (execution succeeded)
Code output:
Scatter plot saved to scatter.png
```

During the generation of response, a human input is requested to give an opportunity to intercept the code execution. In this case, we choose to continue the execution, and the agent’s reply contains the output of the code execution.

We can take a look at the generated plot in the temporary directory.

```python
import os

# We can see the output scatter.png and the code file generated by the agent.
print(os.listdir(temp_dir.name))

# Clean up the working directory to avoid affecting future conversations.
temp_dir.cleanup()
```
```console
['scatter.png', '6507ea07b63b45aabb027ade4e213de6.py']
```

## Docker Execution
To mitigate the security risk of running LLM-generated code locally, we can use the docker command line code executor ([`autogen.coding.DockerCommandLineCodeExecutor`](/docs/api-reference/autogen/coding/DockerCommandLineCodeExecutor)) to execute code in a docker container.

This way, the generated code can only access resources that are explicitly given to it.

The figure below illustrates how docker execution works.

![Code Executor Docker](./assets/code-executor-docker.png)

Similar to the local command line code executor, the docker executor extracts code blocks from input messages, writes them to code files. For each code file, it starts a docker container to execute the code file,
and reads the console output of the code execution.

To use docker execution, you need to [install Docker](https://docs.docker.com/engine/install/) on your machine.
Once you have Docker installed and running, you can set up your code executor agent as follow:

```python
import tempfile
from autogen import ConversableAgent
from autogen.coding import DockerCommandLineCodeExecutor

# Create a temporary directory to store the code files.
temp_dir = tempfile.TemporaryDirectory()

# Create a Docker command line code executor.
executor = DockerCommandLineCodeExecutor(
    image="python:3.12-slim",  # Execute code using the given docker image name.
    timeout=10,  # Timeout for each code execution in seconds.
    work_dir=temp_dir.name,  # Use the temporary directory to store the code files.
    # (Optional): forward any extra kwargs directly to docker.containers.create().
    container_create_kwargs={
        # Example: pass an env-var and request GPU 0
        "environment": {"TOKEN": "<your-token>"},
        # "device_requests": [docker.types.DeviceRequest(device_ids=["0"], capabilities=[["gpu"]])],
    },
)

# Create an agent with code executor configuration that uses docker.
code_executor_agent_using_docker = ConversableAgent(
    "code_executor_agent_docker",
    llm_config=False,  # Turn off LLM for this agent.
    code_execution_config={"executor": executor},  # Use the docker command line code executor.
    human_input_mode="ALWAYS",  # Always take human input for this agent for safety.
)

# When the code executor is no longer used, stop it to release the resources.
# executor.stop()
```

The `work_dir` in the constructor points to a local file system directory just like in the local execution case.
The docker container will mount this directory and the executor write code files and output to it.

### Use Code Execution in Conversation

Writing and executing code is necessary for many tasks such as data analysis, machine learning, and mathematical modeling. In AG2, coding can be a conversation between a code writer agent and a code executor agent, mirroring the interaction between a programmer and a code interpreter.

![Code Writer and Code Executor](./assets/code-execution-in-conversation.png)

The code writer agent can be powered by any LLM with code-writing capability.

And the code executor agent is powered by a code executor.

The following is an agent with a code writer role specified using `system_message`. The system message contains important instruction on how to use the code executor in the code executor agent.

```python

2. When you need to perform some task with code, use the code to perform the task and output the result. Finish the task smartly.
Solve the task step by step if you need to. If a plan is not provided, explain your plan first. Be clear which step uses code, and which step uses your language skill.
When using code, you must indicate the script type in the code block. The user cannot provide any other feedback or perform any other action beyond executing the code you suggest. ## Agentic Code Exploration with Remyx
```

For exploring and executing novel codebases, AG2 supports [Remyx's](https://remyx.ai) Docker images through the [`RemyxCodeExecutor`](/docs/api-reference/autogen/coding/RemyxCodeExecutor). This executor provides instant access to Remyx-built images that reproduce codebases from 7000+ ArXiv papers plus other resources, with dependencies resolved and environments reproduced.

### Key Benefits

- **🚀 Instant Execution**: Pull Remyx-hosted Docker images and run immediately
- **🔍 Searchable Catalog**: Discover implementations with ready-to-use environments
- **🤖 AI-Guided Exploration**: Agents navigate repos, explain code, and run experiments
- **✅ Reproducible**: Codebase replications with version-controlled dependencies
- **🐳 Isolated Containers**: Safe experimentation without environment conflicts
- **⚡ Skip Setup**: Zero dependency resolution or configuration required

### Prerequisites

1. **Get a Remyx API key** from [remyx.ai](https://remyx.ai)
2. **Install Remyx dependencies**:
```bash
   pip install "ag2[remyx]"
```
3. Set your API key:
```bash
export REMYXAI_API_KEY=your_remyx_api_key
```

### Basic Usage
```python
from autogen import ConversableAgent, LLMConfig
from autogen.coding import RemyxCodeExecutor
from remyxai.client.search import SearchClient

# Search for papers with Docker environments
papers = SearchClient().search(
    query="data synthesis techniques",
    has_docker=True,
    max_results=10
)
paper = papers[0]

# Create Remyx executor for the paper
remyx_executor = RemyxCodeExecutor(
    arxiv_id=paper.arxiv_id,
    timeout=300,  # Timeout in seconds
)

# Quick start - AI explores with defaults
remyx_executor.explore()
```

### Configuration Options
The `RemyxCodeExecutor` supports several configuration options:

arxiv_id: ArXiv paper ID (required, e.g., "2301.12345")
api_key: Remyx API key (or set REMYX_API_KEY environment variable)
timeout: Execution timeout in seconds (default: 300)
work_dir: Local directory for code files (optional)

### AI-Guided Exploration
RemyxCodeExecutor includes specialized exploration capabilities through the `explore()` method:
```python
# Quick start - AI explores with defaults
remyx_executor.explore()

# Interactive learning mode
remyx_executor.explore(
    goal="Explain the main finding",
    interactive=True  # Step-by-step with explanations
)

# Batch processing mode
remyx_executor.explore(
    goal="Run a quick test",
    interactive=False  # Automated exploration
)
```
### Use Cases

RemyxCodeExecutor accelerates experimentation and integration:

🔬 **Technique Exploration**: Test novel implementations without setup overhead
⚡ **Rapid Prototyping**: Experiment with cutting-edge methods in minutes
🎓 **Learning**: AI-guided exploration of complex codebases
🔄 **Integration**: Extract and adapt techniques for your projects
📊 **Validation**: Reproduce results in original environments
🧪 **Batch Experimentation**: Run experiments across multiple implementations

### Paper Discovery
Use Remyx's search API to find papers with Docker environments:
```python
from remyxai.client.search import SearchClient

# Search by topic
papers = SearchClient().search(
    "data synthesis strategies",
    has_docker=True,  # Only papers with pre-built environments
    limit=10
)

# Explore each paper
for paper in papers:
    print(f"Paper: {paper.title}")
    print(f"ArXiv ID: {paper.arxiv_id}")
    print(f"Docker available: {paper.has_docker}")

    # Create executor for this paper
    executor = RemyxCodeExecutor(arxiv_id=paper.arxiv_id)
    # ... execute code
```

### How It Works

`RemyxCodeExecutor` provides instant access to working codebases:

* **Search**: Discover Remyx-built Docker images for arXiv papers
* **Instantiate**: Create executor with paper's ArXiv ID or image name
* **Execute**: Run code in isolated containers with dependencies resolved
* **Explore**: Guide AI agents to navigate, explain, and modify code

Remyx-hosted images replicate codebases with all dependencies, CUDA versions, and system libraries pre-installed, eliminating hours of environment setup and manual exploration.

## Serverless Execution with YepCode

For production-grade serverless code execution, AG2 supports [YepCode's](https://yepcode.io) serverless runtime through the [`YepCodeCodeExecutor`](/docs/api-reference/autogen/coding/YepCodeCodeExecutor). This executor runs code in secure, isolated sandboxes with automatic dependency management.

### Key Benefits of YepCode Execution

- **🚀 Serverless**: No infrastructure management required
- **🔒 Secure**: Code runs in isolated, production-grade sandboxes
- **📦 Automatic Dependencies**: NPM and PyPI packages installed automatically
- **🌐 Multi-Language**: Support for Python and JavaScript
- **⚡ Scalable**: Built for enterprise-scale applications
- **☁️ Cloud-Native**: No local setup or Docker required

### Prerequisites

1. **Get a YepCode API token** from [yepcode.io](https://yepcode.io)
2. **Install YepCode dependencies**:
   ```bash
   pip install "ag2[yepcode]"
   ```
3. **Set your API token**:
   ```bash
   export YEPCODE_API_TOKEN=your_yepcode_token
   ```

### Basic Usage

```python
import tempfile
from autogen import ConversableAgent, LLMConfig
from autogen.coding import YepCodeCodeExecutor

# Create YepCode executor
yepcode_executor = YepCodeCodeExecutor(
    timeout=120,           # Timeout in seconds
    remove_on_done=False,  # Keep execution logs for debugging
    sync_execution=True,   # Wait for execution to complete
)

# Create an agent with YepCode executor
code_executor_agent = ConversableAgent(
    "code_executor_agent",
    llm_config=False,  # Turn off LLM for this agent
    code_execution_config={
        "executor": yepcode_executor
    },
    human_input_mode="NEVER",
)

# Test execution
task = """
Run this Python code:
\`\`\`python
import pandas as pd
import numpy as np

# Create sample data with automatic pandas installation
data = pd.DataFrame({
    'x': np.random.randn(100),
    'y': np.random.randn(100)
})

print(f"Generated {len(data)} data points")
print(f"Mean of x: {data['x'].mean():.2f}")
print(f"Mean of y: {data['y'].mean():.2f}")

return {
    "data_points": len(data),
    "x_mean": round(data['x'].mean(), 2),
    "y_mean": round(data['y'].mean(), 2)
}
\`\`\`
"""

reply = code_executor_agent.generate_reply(messages=[{"role": "user", "content": task}])
print(reply)
```

### Factory Pattern

You can also use the factory pattern for convenience:

```python
from autogen.coding import CodeExecutorFactory

# Create YepCode executor using factory
executor = CodeExecutorFactory.create({
    "executor": "yepcode",
    "yepcode": {
        "timeout": 90,
        "remove_on_done": True,  # Clean up after execution
        "sync_execution": True,
    }
})

# Use with agent
agent = ConversableAgent(
    "agent",
    llm_config=False,
    code_execution_config={"executor": executor},
    human_input_mode="NEVER",
)
```

### Configuration Options

The `YepCodeCodeExecutor` supports several configuration options:

- **`api_token`**: YepCode API token (or set `YEPCODE_API_TOKEN` environment variable)
- **`timeout`**: Execution timeout in seconds (default: 60)
- **`remove_on_done`**: Clean up execution after completion (default: False)
- **`sync_execution`**: Wait for execution to complete (default: True)

### Use Cases

YepCode execution is ideal for:

- **🏢 Production Environments**: No Docker or local setup required
- **📊 Data Analysis**: Automatic pandas, numpy, matplotlib installation
- **🌐 API Integration**: Built-in requests, axios availability
- **🔄 CI/CD Pipelines**: Serverless execution in cloud environments
- **👥 Team Collaboration**: Consistent execution environment for all users

### Security and Isolation

YepCode provides enterprise-grade security:

- **Isolated Sandboxes**: Each execution runs in a separate, secure container
- **Network Isolation**: Controlled access to external resources
- **Resource Limits**: CPU and memory limits to prevent resource exhaustion
- **Audit Logging**: Comprehensive execution logs for debugging and compliance

For more examples, see the [YepCode Executor Notebook](/docs/use-cases/notebooks/notebooks/agentchat_yepcode_executor).

## Sandboxed Execution with Daytona

For secure, isolated code execution in the cloud, AG2 supports [Daytona's](https://www.daytona.io) sandbox runtime through the [`DaytonaCodeExecutor`](/docs/api-reference/autogen/coding/DaytonaCodeExecutor). This executor runs code in fast, isolated sandboxes with full support for Python, Bash, JavaScript, and TypeScript.

### Key Benefits

- **🔒 Secure Sandboxes**: Each executor gets its own isolated environment — no LLM-generated code touches your local machine
- **🚀 Fast Startup**: Sandboxes spin up on demand with near-instant startup
- **🐳 Custom Images**: Bring your own Docker image or use Daytona's declarative `Image` builder
- **🌐 Multi-Language**: Python, Bash, sh, JavaScript, TypeScript supported out of the box
- **☁️ Cloud-Native**: No local Docker or infrastructure required
- **♻️ Single Sandbox**: One sandbox reused across all code blocks in a session for efficiency

### Prerequisites

1. **Get a Daytona API key** from [app.daytona.io](https://app.daytona.io/dashboard/keys)
2. **Install Daytona dependencies**:
   ```bash
   pip install "ag2[daytona]"
   ```
3. **Set your API key**:
   ```bash
   export DAYTONA_API_KEY=your_daytona_api_key
   ```

### Basic Usage

```python
from autogen import ConversableAgent, LLMConfig
from autogen.coding import DaytonaCodeExecutor

# Create Daytona executor — sandbox is created immediately
executor = DaytonaCodeExecutor(
    timeout=60,  # Per-execution timeout in seconds
)

# Create an agent with Daytona executor
code_executor_agent = ConversableAgent(
    "code_executor_agent",
    llm_config=False,
    code_execution_config={"executor": executor},
    human_input_mode="NEVER",
)
```

Or use it as a context manager for automatic sandbox cleanup:

```python
with DaytonaCodeExecutor(timeout=60) as executor:
    code_executor_agent = ConversableAgent(
        "code_executor_agent",
        llm_config=False,
        code_execution_config={"executor": executor},
        human_input_mode="NEVER",
    )
    # Sandbox is automatically deleted when the with-block exits
```

### Snapshots

A Daytona **snapshot** is a pre-configured environment template registered in your Daytona account:

```python
executor = DaytonaCodeExecutor(snapshot="my-data-science-snapshot", timeout=60)
```

Snapshots take priority over `image` — you cannot set both.

### Custom Images

Bring a custom Docker image using a plain string or Daytona's declarative `Image` builder:

```python
from autogen.coding import DaytonaCodeExecutor, DaytonaSandboxResources
from daytona import Image

# Plain image string
executor = DaytonaCodeExecutor(image="python:3.12-slim", timeout=60)

# Declarative image with pre-installed packages
custom_image = Image.base("python:3.12-slim").pip_install("pandas", "numpy", "matplotlib")
executor = DaytonaCodeExecutor(image=custom_image, timeout=120)

# With resource limits (only applied when using a custom image)
executor = DaytonaCodeExecutor(
    image=custom_image,
    resources=DaytonaSandboxResources(cpu=2, memory=4, disk=4),
    timeout=120,
)
```

### Configuration Options

- **`api_key`**: Daytona API key (or set `DAYTONA_API_KEY` environment variable)
- **`api_url`**: Daytona API URL (or set `DAYTONA_API_URL` environment variable)
- **`target`**: Target region e.g. `"us"`, `"eu"` (or set `DAYTONA_TARGET` environment variable)
- **`timeout`**: Per-execution timeout in seconds (default: 60)
- **`snapshot`**: Named Daytona snapshot to use as the sandbox base
- **`image`**: Custom Docker image — string name or declarative `Image` object
- **`name`**: Human-readable sandbox name (auto-generated if not set)
- **`env_vars`**: Environment variables available inside the sandbox (useful for passing secrets to code)
- **`resources`**: CPU/memory/disk limits via `DaytonaSandboxResources` (only applied with custom images)

### Use Cases

Daytona execution is ideal for:

- **🔒 Security-First Environments**: LLM-generated code runs in isolated cloud sandboxes, not on your machine
- **📦 Custom Environments**: Pre-install packages via declarative `Image` builds
- **🌐 Multi-Language Tasks**: Mix Python, Bash, JavaScript, and TypeScript in the same session
- **☁️ Cloud Workflows**: No local Docker or infrastructure setup required
- **🔑 Secret Injection**: Pass API keys and credentials via `env_vars` without hardcoding them in code

#### Command Line or Jupyter Code Executor?

The command line code executor does not keep any state in memory between executions of different code blocks it receives, as it writes each code block to a separate file and executes the code block in a new process.

Contrast to the command line code executor, the Jupyter code executor runs all code blocks in the same Jupyter kernel, which keeps the state in memory between executions.

The choice between command line and Jupyter code executor depends on the nature of the code blocks in agents' conversation. If each code block is a "script" that does not use variables from previous code blocks, the command line code executor is a good choice. If some code blocks contain expensive computations (e.g., training a machine learning model and loading a large amount of data), and you want to keep the state in memory to avoid repeated computations, the Jupyter code executor is a better choice.
